【项目描述】                                                                    
这是一个用于记录朋友聚会的 Web 应用（类似在线日记 +                             
数据统计平台），支持记录聚会、生成日记、图片管理和数据分析。                    
                                                                                
【核心功能】                                                                    
1. 用户记录聚会（时间、地点、餐厅、酒类、主题、参与人）                         
2. 数据统计：                                                                   
   - 每季度聚会次数（图表展示）                                                 
   - 每人参与次数排行榜（显示第一名和最后一名）                                 
3. 日记系统：                                                                   
   - 每次聚会自动生成一个日记页面                                               
   - 支持图片集 + 图片描述                                                      
   - 支持评论互动                                                               
                                                                                
【你的任务（必须按顺序输出）】                                                  
请严格分阶段输出，不要跳步：

Phase 6：开发路线图（适合个人开发者）                                           
Phase 7：风险分析 + 简化建议

  MVP 定义

  包含： 聚会 CRUD、朋友管理、自动日记、图片上传、季度图表、排行榜、基础登录

  排除： 评论、多用户权限、通知

  理由： 核心价值是"记录 + 回忆"，社交互动是增值功能。先让数据能存进去、看出来。

  推荐：轻量全栈（Supabase + Next.js + Vercel 三件套）

  前端 (Next.js 14 App Router)
    ↕ Server Actions / API Routes
  数据库 (PostgreSQL via Supabase)
    +
  对象存储 (Supabase Storage)

  ├────────┼───────────────────────┼────────────────────────────────────┤
  │ 框架   │ Next.js 14 App Router │ SSR + API 合一，一个项目搞定前后端 │
  ├────────┼───────────────────────┼────────────────────────────────────┤

  ├────────┼───────────────────────┼────────────────────────────────────┤
  │ 图表   │ Recharts              │ 轻量，React 原生                   │

  理由： 个人开发首选"一栈到底"，可在 1 天内完成基础搭建，且全程免费。

  ---
  Phase 3：数据模型设计

  实体关系

  User → (一对多) → Gathering → (一对一) → DiaryEntry
  Friend → (多对多，通过关联表) → Gathering
  Gathering → (一对多) → Photo
  DiaryEntry → (一对多) → Comment (V2)

  核心表结构

  gatherings（聚会）
  - id, user_id, title, held_at, location, restaurant, drinks(数组), notes

  friends（朋友）
  - id, user_id, name, avatar_url

  gathering_participants（多对多关联）
  - gathering_id, friend_id

  diary_entries（日记，与 gathering 一对一）
  - id, gathering_id, content, updated_at

  photos（图片）
  - id, gathering_id, url, caption, sort_order

  设计重点：
  - drinks 用 PostgreSQL 数组，避免过度建表
  - 创建聚会时自动同步创建空日记记录
  - 不建独立评论用户表（V2 改进前用 author_name 字符串）

  ---
  Phase 4：API 设计

  接口规范（RESTful，Next.js API Routes）

  # 聚会
  GET/POST       /api/gatherings
  GET/PUT/DELETE /api/gatherings/:id

  # 朋友
  GET/POST       /api/friends
  PUT/DELETE     /api/friends/:id

  # 日记
  GET/PUT        /api/diary/:gathering_id

  # 图片
  POST           /api/photos/upload
  PUT/DELETE     /api/photos/:id

  # 统计
  GET            /api/stats/quarterly    ?year=2024
  GET            /api/stats/participants

  通用约定：
  - 响应格式：{ data, error, meta }
  - 认证：Supabase JWT Bearer Token
  - 错误码：标准 HTTP

  理由： 统计接口独立便于缓存；MVP 无需 GraphQL，REST 足够。

  │         路由          │               页面               │
  ├───────────────────────┼──────────────────────────────────┤

  ├───────────────────────┼──────────────────────────────────┤
  │ /gatherings/new       │ 创建聚会表单                     │
  ├───────────────────────┼──────────────────────────────────┤
  │ /gatherings/:id       │ 聚会详情                         │
  ├───────────────────────┼──────────────────────────────────┤
  │ /gatherings/:id/diary │ 日记页（图片集 + 正文）          │
  ├───────────────────────┼──────────────────────────────────┤
  │ /friends              │ 朋友管理                         │
  ├───────────────────────┼──────────────────────────────────┤
  │ /stats                │ 完整统计页                       │
  └───────────────────────┴──────────────────────────────────┘

  关键设计决策

  - Server Components（默认）：列表、详情、统计页 → 直连 DB，首屏快
  - Client Components：表单、图片上传、图表 → 需要交互才降级
  - 状态管理：无需 Redux，用 React Server Actions + useOptimistic 足够

  ---
  Phase 7：风险分析 + 简化建议

  主要风险

  ┌─────────────────────────┬──────┬───────────────────────────────────┐
  │          风险           │ 等级 │               应对                │
  ├─────────────────────────┼──────┼───────────────────────────────────┤
  │ 功能蔓延（Scope Creep） │ 高   │ 严格按 MVP，V2 功能进 backlog     │
  ├─────────────────────────┼──────┼───────────────────────────────────┤
  │ 图片上传体验差          │ 中   │ 用 react-dropzone，限制 < 5MB     │
  ├─────────────────────────┼──────┼───────────────────────────────────┤
  │ DB 改动频繁             │ 中   │ Prisma migration 管理，不手动改表 │
  ├─────────────────────────┼──────┼───────────────────────────────────┤
  │ 图片存储费用            │ 低   │ Supabase 免费 1GB，压缩后上传     │
  └─────────────────────────┴──────┴───────────────────────────────────┘

  立即可砍的复杂度（节省 ~30% 工作量）

  1. 评论功能 → 延后到 V2（MVP 没人需要）
  2. 日记编辑 → 普通 Textarea，不引入富文本编辑器
  3. 朋友头像 → 首字母 Avatar，不做头像上传
  4. 统计图表 → 简单 BarChart，不做交互动画

     Phase 1：产品功能拆解

     模块划分

     ┌────────────┬─────────────────────────────┬────────┐
     │    模块    │            功能             │ 优先级 │
     ├────────────┼─────────────────────────────┼────────┤
     │ 聚会记录   │ 创建/编辑/删除聚会事件      │ MVP    │
     ├────────────┼─────────────────────────────┼────────┤
     │ 参与人管理 │ 维护朋友列表                │ MVP    │
     ├────────────┼─────────────────────────────┼────────┤
     │ 日记系统   │ 每次聚会自动生成日记页面    │ MVP    │
     ├────────────┼─────────────────────────────┼────────┤
     │ 图片管理   │ 上传/展示/描述图片集        │ MVP    │
     ├────────────┼─────────────────────────────┼────────┤
     │ 数据统计   │ 季度聚会图表 + 参与人排行   │ MVP    │
     ├────────────┼─────────────────────────────┼────────┤
     │ 评论互动   │ 对日记页面发表评论          │ V2     │
     ├────────────┼─────────────────────────────┼────────┤
     │ 用户认证   │ 登录/注册（或简单密码保护） │ MVP    │
     └────────────┴─────────────────────────────┴────────┘

     MVP 定义（最小可上线版本）

     包含：
     - 聚会事件 CRUD（时间、地点、餐厅、酒类、主题、参与人）
     - 朋友列表管理
     - 自动生成日记页面（含图片上传）
     - 季度聚会次数图表
     - 参与人次数排行榜（显示第一、最后）
     - 基础身份验证（单用户或简单密码）

     排除 MVP：
     - 评论系统（V2）
     - 多用户权限（V2）
     - 通知/提醒（V3）

     设计理由： 核心价值是"记录 +
     回忆"，统计是附加价值，评论是社交功能。优先让数据可记录、可查看。

     ---
     Phase 2：系统架构设计

     推荐架构：轻量全栈（个人开发者最优解）

     前端 (Next.js 14 App Router)
       ↕ HTTP / Server Actions
     后端 (Next.js API Routes 或 Server Actions)
       ↕
     数据库 (PostgreSQL via Supabase)
       +
     对象存储 (Supabase Storage 或 Cloudflare R2)

     技术选型

     ┌──────────┬──────────────────────────┬───────────────────────────────────┐
     │   层级   │           技术           │               理由                │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ 前端框架 │ Next.js 14 (App Router)  │ SSR + API 合一，减少部署复杂度    │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ UI 组件  │ shadcn/ui + Tailwind CSS │ 美观、开箱即用、无需设计师        │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ 图表     │ Recharts                 │ 轻量，React 原生支持              │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ 数据库   │ PostgreSQL (Supabase)    │ 免费额度足够，内置 Auth + Storage │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ 图片存储 │ Supabase Storage         │ 与 DB 同平台，省去跨服务配置      │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ 认证     │ Supabase Auth (或        │ 开箱即用，支持邮箱/社交登录       │
     │          │ NextAuth)                │                                   │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ 部署     │ Vercel                   │ Next.js                           │
     │          │                          │ 最优部署平台，免费额度够用        │
     ├──────────┼──────────────────────────┼───────────────────────────────────┤
     │ ORM      │ Prisma 或 Drizzle ORM    │ 类型安全，迁移管理方便            │
     └──────────┴──────────────────────────┴───────────────────────────────────┘

     设计理由： 单人开发首选"一栈到底"方案，Supabase + Next.js + Vercel
     三件套可在一天内完成基础搭建，且免费额度对个人项目完全够用。

     Phase 3：数据模型设计

     实体关系概览

     User (用户)
       └─ 一对多 → Gathering (聚会)

     Friend (朋友)
       └─ 多对多 → Gathering (通过 GatheringParticipant)

     Gathering (聚会)
       ├─ 一对一 → DiaryEntry (日记)
       └─ 一对多 → Photo (图片)

     DiaryEntry (日记)
       └─ 一对多 → Comment (评论, V2)

     Phase 4：API 设计

     接口规范（RESTful，基于 Next.js API Routes）

     聚会管理

     GET    /api/gatherings              获取聚会列表（支持分页、年份过滤）
     POST   /api/gatherings              创建聚会（同时创建 diary_entry）
     GET    /api/gatherings/:id          获取聚会详情（含参与人、图片）
     PUT    /api/gatherings/:id          更新聚会
     DELETE /api/gatherings/:id          删除聚会

     朋友管理

     GET    /api/friends                 获取朋友列表
     POST   /api/friends                 添加朋友
     PUT    /api/friends/:id             更新朋友信息
     DELETE /api/friends/:id             删除朋友

     日记

     GET    /api/diary/:gathering_id     获取聚会日记（含评论）
     PUT    /api/diary/:gathering_id     更新日记内容

     图片

     POST   /api/photos/upload           上传图片（multipart，返回 URL）
     PUT    /api/photos/:id              更新图片描述/排序
     DELETE /api/photos/:id              删除图片

     统计

     GET    /api/stats/quarterly         季度聚会次数（?year=2024）
     GET    /api/stats/participants      参与人排行榜（含总次数）

     评论（V2）

     GET    /api/comments/:diary_id      获取评论列表
     POST   /api/comments/:diary_id      发表评论
     DELETE /api/comments/:id            删除评论

     通用约定

     - 响应格式：{ data, error, meta }
     - 分页：?page=1&limit=20
     - 错误码遵循 HTTP 标准（400/401/404/500）
     - 认证：Bearer Token（Supabase JWT）

     设计理由： 统计接口独立设计便于缓存；图片上传独立接口便于限速控制；MVP
     阶段无需 GraphQL，REST 足够简洁。

         ├── PhotoUploader             # 拖拽上传

         ├── ParticipantRanking        # 排行榜组件
         └── DiaryEditor               # 富文本或 Markdown 编辑器

     数据流设计

     - Server Components（默认）：列表页、详情页、统计页（SEO 友好，直接读 DB）
     - Client Components：表单、图片上传、图表（需要交互）
     - 数据获取：优先 Server Components 直接调用 DB（通过 Prisma），减少 API
     请求
     - 状态管理：无需 Redux，用 React Server Actions + useOptimistic
     处理乐观更新

     设计理由： App Router 的 Server Components
     让列表/详情页无需客户端请求，首屏速度快；表单和上传才降级为 Client
     Components，保持架构简洁。

     Phase 6：开发路线图

     总体原则：按"可展示里程碑"切分，每周有可看产出

     Week 1：项目基础 + 认证

     - 初始化 Next.js 14 项目（TypeScript + Tailwind + shadcn/ui）
     - 配置 Supabase（数据库 + Auth + Storage）
     - 定义 Prisma Schema + 执行首次 migration
     - 实现登录/登出页面
     - 配置 Vercel 部署（持续集成）

     Week 2：核心功能（聚会 + 朋友）

     - 朋友列表 CRUD 页面
     - 聚会创建表单（含参与人多选）
     - 聚会列表页（时间线展示）
     - 聚会详情页

     Week 3：日记 + 图片

     - 图片上传（Supabase Storage）
     - 日记页面（图片网格 + 描述编辑）
     - 日记正文编辑（Markdown 或简单 Textarea）
     - 聚会创建时自动生成日记记录

     Week 4：数据统计

     - 季度聚会次数图表（Recharts BarChart）
     - 参与人次数排行榜
     - Dashboard 统计概览
     - 年份切换过滤

     Week 5：打磨 + V2 起步

     - 响应式适配（移动端）
     - 加载状态、错误处理
     - 评论功能（V2）
     - 图片拖拽排序

     设计理由： 每周交付一个可用模块，避免"全做完再上线"导致拖期。Week 1
     就部署到 Vercel，之后持续迭代。

     Phase 7：风险分析 + 简化建议

     风险项

     ┌───────────────────────┬──────┬──────────────────────────────────────────┐
     │         风险          │ 等级 │                 应对方案                 │
     ├───────────────────────┼──────┼──────────────────────────────────────────┤
     │ 图片存储费用超支      │ 低   │ Supabase Storage 免费                    │
     │                       │      │ 1GB，压缩上传前的图片                    │
     ├───────────────────────┼──────┼──────────────────────────────────────────┤
     │ 数据库设计改动频繁    │ 中   │ 用 Prisma Migration                      │
     │                       │      │ 管理变更，避免手动改表                   │
     ├───────────────────────┼──────┼──────────────────────────────────────────┤
     │ 认证系统复杂度        │ 低   │ 直接用 Supabase Auth，不自研             │
     ├───────────────────────┼──────┼──────────────────────────────────────────┤
     │ 功能蔓延（Scope       │ 高   │ 严格按 MVP 执行，V2 功能记入 backlog     │
     │ Creep）               │      │                                          │
     ├───────────────────────┼──────┼──────────────────────────────────────────┤
     │ 图片上传体验差        │ 中   │ 用成熟库（react-dropzone），限制单张 <   │
     │                       │      │ 5MB                                      │
     ├───────────────────────┼──────┼──────────────────────────────────────────┤
     │ 单用户 vs 多用户      │ 中   │ MVP 只做单用户（自己用），V2             │
     │                       │      │ 再加邀请机制                             │
     └───────────────────────┴──────┴──────────────────────────────────────────┘

     简化建议（可减少 30%+ 开发量）

     1. 评论功能延后：MVP 完全不做，上线后看是否真的需要
     2. 跳过富文本编辑器：日记用普通 Textarea 或 Markdown，避免引入重量级依赖
     3. 图片描述内联编辑：直接在图片下方 click-to-edit，无需单独编辑页面
     4. 统计图表简化：先做简单 BarChart，不做可交互的复杂图表
     5. 朋友头像暂缓：初期只显示姓名首字母 Avatar，无需上传头像

     最终技术栈确认

     Next.js 14 (App Router)
     + TypeScript
     + Tailwind CSS + shadcn/ui
     + Prisma ORM
     + Supabase (PostgreSQL + Auth + Storage)
     + Recharts
     + Vercel (部署)

     设计理由： 这套组合是 2024-2025 年个人全栈项目的黄金组合——类型安全、部署简
     单、免费额度充足、生态成熟。无需额外的后端服务器，一个人可以独立维护。
